# Architecture

This document describes the high-level architecture of AI Medical Assist.

## High-Level System

The platform is a three-tier setup:

1. React frontend (`frontend/`) served over HTTP.
2. FastAPI backend (`backend/`) exposing `/api/*` endpoints.
3. PostgreSQL database (in Docker Compose) for persistent application data.

Optional external services are used by the backend:

- Groq for language model responses and report analysis.
- Deepgram for speech-to-text and text-to-speech.

## Deployment Topology

`docker-compose.yml` defines:

- `frontend` container (Nginx + built Vite app), exposed on `8080`.
- `backend` container (FastAPI), exposed on `8000`.
- `db` container (PostgreSQL), exposed on `5432`.

Named volumes:

- `pgdata` for Postgres persistence.
- `backend_data` for uploads/reports generated by backend.

## Backend Architecture

Backend entrypoint: `backend/app/main.py`.

### Key Layers

- **Routers** (`backend/app/routers`): HTTP endpoint definitions grouped by domain.
- **Dependencies/Security**: auth, role checks, JWT handling.
- **Agents** (`backend/app/agents`): routing and specialist prompt orchestration.
- **Services** (`backend/app/services`): integrations and business flows (Groq, Deepgram, document extraction, PDFs).
- **Persistence**: SQLAlchemy models + sessions + migrations.

### Router Domains

- `auth`: registration/login/current user.
- `chat`: text chat sessions, message history, AI chat turn.
- `voice`: audio turn (transcribe -> AI reply -> synthesize).
- `reports`: report generation and uploaded report analysis.
- `labs`: analyzed lab result retrieval and PDF access.
- `appointments`: specialties, doctor profiles, slots, booking.
- `vitals`: patient vitals.
- `doctor_portal`: doctor-facing workflows.
- `admin`: administrative capabilities.

### Agent Orchestration Flow

Main orchestrator: `backend/app/agents/orchestrator.py`.

For each chat turn:

1. Inspect recent messages and latest user text.
2. Apply urgent/safety prefix heuristics when needed.
3. Select specialist route:
   - manual (`agent_mode`) or
   - auto route via router agent.
4. Build specialist system prompt + guardrails.
5. Call Groq chat completion with composed prompt and conversation context.
6. Return assistant response and trace metadata (`agent_trace`).

## Voice Pipeline

`/api/voice/turn` flow:

1. Frontend records browser audio and sends base64 payload.
2. Backend decodes and transcribes via Deepgram STT.
3. Transcribed text is sent through agent orchestrator.
4. Reply is synthesized to audio with Deepgram TTS.
5. Response includes transcript, reply text, optional base64 audio, and trace.
6. Backend creates a report entry from the interaction summary.

## Report Analysis Pipeline

Upload analysis path (`/api/reports/analyze-upload`):

1. Store uploaded file in `UPLOADS_DIR`.
2. Extract text:
   - native PDF/text parsing when possible.
   - OCR fallback for scanned PDFs.
3. Analyze extracted content with LLM report flow.
4. Generate analysis PDF from markdown.
5. Persist metadata and analysis in `LabResult`.
6. Return markdown + id for frontend rendering/download.

## Data and Storage

Backend creates/uses local directories:

- `UPLOADS_DIR` (raw uploaded files).
- `REPORTS_DIR` (generated report PDFs).

Database stores:

- users and roles,
- chat sessions/messages,
- appointments, doctor profiles, slots,
- vitals,
- generated medical reports,
- lab report analyses and PDF paths.

## Security Model

- JWT-based authentication for API calls.
- Role-based access control (patient/doctor/admin) in route dependencies.
- Resource-level checks for sensitive downloads (for example report PDF access by linked patient/doctor/admin only).

## Startup Behavior

On backend startup:

- Database tables are initialized.
- a migration hook for lab result PDF column is executed.
- upload/report directories are created.
- optional admin user is seeded from environment variables.
- demo doctors and future appointment slots are seeded.
